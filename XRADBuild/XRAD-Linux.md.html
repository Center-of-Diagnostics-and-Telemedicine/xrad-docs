**Сборка XRAD в Linux** # Введение # Способ сборки библиотеки XRAD и приложений требует продумывания и доработки. Как требуют доработки и скрипты CMake, использующиеся для сборки. Здесь описан первый вариант способа сборки. # Условные обозначения # `/home/{user_name}/` — домашняя директория пользователя, `~`. Используется в тех местах, где путь, начинающийся с `~/`, трактуется неверно. # Инструменты # OS: Ubuntu 18.04 (WSL). GCC: версия 9.2.1. CMake: использовался CMake 3.10.2 (установлено из пакетов). Библиотеки: см. [Список сторонних продуктов](../3rd_party/Third_party.md.html). Замечание. CMake работает неверно, если в качестве пути директории для сборки (binaries) указывать путь, начинающийся с `~`. Путь должен быть абсолютным: `/home/{user_name}/...`, где `/home/{user_name}` — путь к домашней директории пользователя (`~`). То же относится ко многим другим путям, задаваемым в переменных CMake. Возможно, дело не в само́м CMake, а в скриптах. Замечание. В инструкции использована графическая оболочка `cmake-gui`. Использование графической оболочки не является обязательным. Все операции можно выполнить с командной строки командой `cmake`. Замечание. Вместо команды `cmake --build .` можно использовать просто `make`. # Принцип работы системы сборки # Здесь описана схема, использованная на данный момент. См. также раздел [TODO]. ## Экспортируемые символы (функции, переменные) ## По умолчанию gcc при компиляции и линковке создает все функции и переменные экспортируемыми. Все, включая шаблоны C++. В результате при сборке модуля динамической библиотеки (например, libFantomLibrary.so) у неё получается огромного размера таблица экспортируемых символов. Помимо того, что это некрасиво, это замедляет загрузку такого модуля. Есть несколько способов решить эту проблему. Наиболее правильным является следующий. В CMake-файлах задаем следующие опции: ~~~~ none set(CMAKE_CXX_VISIBILITY_PRESET hidden) set(CMAKE_VISIBILITY_INLINES_HIDDEN 1) ~~~~ Для исходников на C нужно задать: ~~~~ none set(CMAKE_C_VISIBILITY_PRESET hidden) ~~~~ Для сторонних статических библиотек эти переменные могут быть заданы через интерфейс CMake (в `cmake-gui` — кнопкой "Add Entry"). Для сторонних библиотек, допускающих использование достаточно старых версий CMake, выдающих при этих значениях переменных ошибку на стадии Generate, также потребуется в CMake-файл добавить инструкцию: ~~~~ none cmake_policy(SET CMP0063 NEW) ~~~~ В результате по умолчанию все функции и переменные будут не экспортируемыми. Чтобы экспортировать выбранные функции, для них при компиляции нужно задать атрибут gcc `__attribute__((visibility("default")))`. Это аналогично заданию `__declspec(dllexport)` в MSVC (атрибут, аналогичный `__declspec(dllimport)`, в gcc не требуется). Пример определения макроса с атрибутом в FantomLibrary: ~~~~ c++ #ifndef FANTOM_DLL_EI #if defined(XRAD_COMPILER_MSC) #ifdef Fantom_DLL #define FANTOM_DLL_EI __declspec(dllexport) #else #define FANTOM_DLL_EI __declspec(dllimport) #endif #elif defined(XRAD_COMPILER_GNUC) #ifdef Fantom_DLL #define FANTOM_DLL_EI __attribute__((visibility("default"))) #else #define FANTOM_DLL_EI #endif #else #define FANTOM_DLL_EI #endif //_MSC_VER #endif extern "C" int FANTOM_DLL_EI ExportedFunction(); ~~~~ Символ `Fantom_DLL` должен быть определен при сборке проекта FantomLibrary и не определен при сборке других проектов. ## Линковка динамических библиотек so ## Если в CMake-файле приложения в `target_link_libraries` передать имя динамической библиотеки so с путем, то при загрузке приложения системный загрузчик будет искать эту библиотеку по указанному пути. Это удобно в Debug-сборках, библиотеку не надо никуда копировать. Но для Release-сборок это нехорошо. Есть способы управления поиском загружаемых библиотек. В Интернете предлагается несколько вариантов: - Задавать имя библиотеки без пути, а путь передавать в списке путей поиска библиотек при линковке. Тогда при запуске приложения будет использоваться стандартная процедура поикса динамических библиотек. Следует иметь в виду, что в Linux динамические библиотеки по умолчанию ищутся только в системных директориях, в директории приложения и в текущей директории поиск не производится (см. статьи по ключевому слову `LD_LIBRARY_PATH`). - Задать специфические для gcc флаги линкера. Это позволяет производить тонкую настройку путей поиска библиотек для приложения. См. ключ GCC `-rpath`.
